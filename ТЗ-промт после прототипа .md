# ТЗ: Симулятор развития навыков CPO в IT-корпорации (TypeScript MVP)

## Контекст

Нужно реализовать веб-приложение «Симулятор развития навыков CPO» — интерактивный тренажёр для менеджеров продуктов в крупной IT-корпорации.  
Приложение состоит из двух связанных разделов:

1. Диагностика продуктовых навыков
2. Симулятор продуктовых навыков

Зафиксированный HTML-прототип является основным визуальным референсом для дальнейшей реализации MVP.  
Все последующие этапы разработки должны опираться на этот UI-бейслайн, если иное не будет согласовано отдельно.

## Статус визуального решения

Последняя утверждённая HTML-версия считается:
- базовым экранным решением MVP;
- эталоном структуры интерфейса;
- отправной точкой для реализации на TypeScript + Vite.

Это означает:
- тёмная тема используется как основная;
- авторизация на этапе MVP отсутствует;
- отдельный home-экран не требуется;
- приложение стартует сразу в основном интерфейсе;
- слева находятся 2 верхнеуровневых раздела;
- внутри каждого раздела есть собственные шаги, отображаемые в демонстрационном окне;
- зоны загрузки файлов закреплены внизу соответствующего окна.

## Цель MVP

Собрать работающий браузерный MVP, который:
- запускается как статический фронтенд;
- работает внутри корпоративной среды;
- не требует backend;
- не использует localStorage/sessionStorage;
- использует заглушки вместо реального AI;
- архитектурно готов к замене заглушек на реальные интеграции позднее.

## Технологический стек

### Основной стек
- TypeScript как основной язык
- Vite как инструмент сборки и локальной разработки
- HTML + CSS + TypeScript без тяжёлых UI-фреймворков
- SheetJS (`xlsx`) для чтения Excel `.xlsx`
- mammoth.js для чтения Word `.docx`

### Уточнение по стеку
- TypeScript — обязательное требование
- Vite — обязательный инструмент сборки/dev-среды
- React/Vue не использовать, если они не понадобятся отдельно и не будут согласованы

### Формат поставки
На выходе должен быть проект, который:
1. разрабатывается как TypeScript-приложение через Vite;
2. собирается командой `npm run build`;
3. выдаёт готовую папку `dist/` для поставки как статический фронтенд.

## Общие принципы разработки

1. Код писать как основу для роста продукта
2. Все ключевые сущности типизировать через `type`/`interface`
3. Весь UI-текст — на русском языке
4. Все ошибки файлов, парсинга и валидации — понятны пользователю
5. Архитектура должна допускать замену MVP-заглушек на реальные интеграции
6. Не использовать `localStorage` и `sessionStorage`
7. Не реализовывать backend в MVP
8. Не усложнять архитектуру без необходимости
9. Текущий утверждённый HTML служит референсом структуры экранов и базовой визуальной иерархии

## Сценарий продукта

Приложение содержит два основных раздела.

### Раздел 1 — Диагностика продуктовых навыков
Пользователь:
- загружает Excel с критериями;
- загружает Word с кейсом и заданиями;
- проходит 11 заданий;
- вводит ответы в текстовом виде;
- видит экран анализа;
- получает результаты, комментарии и рекомендации;
- выбирает навыки для изучения в симуляторе.

### Раздел 2 — Симулятор продуктовых навыков
Пользователь:
- загружает Word с контентом симулятора;
- видит список навыков;
- выбирает один из навыков;
- проходит обучение по линейной последовательности блоков:
  - теория,
  - кейс,
  - задание,
  - обратная связь,
  - следующий блок;
- получает итог по выбранному навыку.

## Целевая аудитория

- Менеджеры продуктов внутри крупной корпорации
- Уровень: знают базовую теорию product management, имеют частичную практику
- Цель: безопасно тренировать мышление и прикладные навыки

## Ограничения MVP

### Что входит в MVP
- Зафиксированный UI на основе последнего HTML-прототипа
- 2 верхнеуровневых раздела:
  - Диагностика продуктовых навыков
  - Симулятор продуктовых навыков
- Полный пользовательский сценарий диагностики
- Полный пользовательский сценарий симулятора
- Загрузка Excel и Word-файлов через UI
- Парсинг файлов и работа по их содержимому
- Заглушка AI-оценки в диагностике
- Выбор рекомендованных навыков
- Переход из диагностики в симулятор
- Итоговая оценка по прохождению навыка
- SPA-подобная внутренняя навигация без backend-routing

### Что не входит в MVP
- Реальная авторизация
- Визуальная авторизация-заглушка
- Отдельный home-экран
- Сохранение прогресса между сессиями
- Production-ready безопасность
- Backend / БД / API-сервер
- Реальный AI API
- Корпоративный брендинг
- Полноценная мобильная адаптация

## Навигационная модель приложения

Приложение должно быть реализовано как SPA-подобный интерфейс.

### Верхний уровень
Слева пользователь видит 2 раздела:
- `diagnostic`
- `simulator`

### Внутренние шаги раздела diagnostic
Шаги отображаются внутри одного экранного окна:
- `diagnostic-intro`
- `diagnostic-task`
- `diagnostic-processing`
- `diagnostic-results`

Зона загрузки файлов диагностики всегда отображается внизу окна раздела.

### Внутренние шаги раздела simulator
Шаги отображаются внутри одного экранного окна:
- `simulator-skills`
- `simulator-session`
- `simulator-results`

Зона загрузки файла симулятора всегда отображается внизу окна раздела.

### Уточнение по структуре
Отдельные экраны `auth`, `home`, `diagnostic-upload`, `simulator-upload` как самостоятельные полноэкранные состояния больше не нужны.  
Их логика сохраняется, но визуально встроена внутрь двух основных разделов.

## Раздел 1: Диагностика продуктовых навыков

### Встроенная зона загрузки
Внизу окна диагностики должны быть:
- загрузка Excel с критериями;
- загрузка Word с кейсом и заданиями;
- индикация успешной загрузки;
- понятные ошибки.

### Шаг `diagnostic-intro`
Показать:
- название раздела;
- краткое объяснение процесса;
- количество заданий;
- вводную часть кейса;
- кнопку старта.

### Шаг `diagnostic-task`
Показать:
- прогресс `Задание X из 11`;
- название навыка;
- текст задания;
- поле ответа;
- счётчик символов;
- навигационные кнопки.

Требования:
- ответы хранятся в AppState;
- ответ восстанавливается при возврате;
- минимум 50 символов;
- переход вперёд блокируется, если условие не выполнено.

### Шаг `diagnostic-processing`
Показать:
- сообщение анализа;
- анимированный прогресс;
- задержку 3–5 секунд.

Логику вынести в `diagnosticEvaluator.ts`.

Функция:
```ts
export async function evaluateDiagnosticAnswers(
  request: AiEvaluationRequest
): Promise<AiEvaluationResponse>
```

В коде предусмотреть:
- `TODO: заменить на корпоративный AI API`
- `TODO: добавить калибровку оценок`

### Шаг `diagnostic-results`
Показать:
- оценки по навыкам;
- комментарии;
- блок рекомендаций;
- блок дополнительных навыков;
- чекбоксы выбора навыков;
- кнопку перехода в симулятор.

Правило рекомендаций:
- 1–3 → рекомендовать;
- 4–6 → можно изучить дополнительно.

## Раздел 2: Симулятор продуктовых навыков

### Встроенная зона загрузки
Внизу окна симулятора:
- загрузка Word-файла с контентом;
- индикация успешной загрузки;
- ошибки структуры/парсинга.

### Шаг `simulator-skills`
Показать:
- карточки навыков;
- выделение рекомендованных;
- статусы;
- текст-заглушку о будущем сохранении прогресса.

### Шаг `simulator-session`
Организовать линейный сценарий:
1. Теория
2. Кейс
3. Задание
4. Обратная связь
5. Следующий блок

Поддержать блоки:
- `theory`
- `case-description`
- `choice-question`
- `open-question`
- `feedback`

Правила:
- для `choice-question` нельзя идти дальше без выбора;
- для `open-question` минимум 30 символов;
- прогресс навыка всегда виден сверху.

### Шаг `simulator-results`
Показать:
- название навыка;
- процент/оценку;
- список пройденных заданий и статусов;
- возврат к навыкам;
- возврат к результатам диагностики.

## Состояние приложения

В `appState.ts` хранить:
- текущий пользователь;
- текущий верхнеуровневый раздел;
- текущий внутренний шаг диагностики;
- текущий внутренний шаг симулятора;
- загруженные диагностические данные;
- ответы диагностики;
- результаты диагностики;
- выбранные навыки;
- данные симулятора;
- текущий навык;
- прогресс симулятора;
- результаты по навыкам.

Изменение состояния — только через явные методы/функции.

## Обработка ошибок

Нужно покрыть сценарии:
- неверный тип файла;
- отсутствуют обязательные колонки в Excel;
- нарушена структура Word;
- пустой файл;
- не удалось распарсить файл;
- количество заданий в Excel и Word не совпадает;
- пользователь пытается продолжить без обязательного ответа.

Ошибки показывать простым русским языком.

## Качество кода

### Обязательно
- TypeScript `strict`
- без `any`, кроме вынужденных мест с пояснением
- небольшие функции с одной ответственностью
- повторяющуюся UI-логику выносить
- не смешивать парсинг, бизнес-логику и рендеринг
- публичные функции кратко документировать
- понятные названия сущностей

### Точки расширения
- подключение AI API
- подключение авторизации
- подключение сохранения прогресса
- замена файловых источников на backend/API

## Финальный сценарий MVP

Топ-менеджер должен иметь возможность:
1. открыть приложение;
2. перейти в диагностику;
3. загрузить Excel и Word по диагностике;
4. пройти 11 заданий;
5. получить оценки и рекомендации;
6. выбрать навыки для изучения;
7. перейти в симулятор;
8. загрузить контент симулятора;
9. выбрать навык;
10. пройти его;
11. получить итоговую оценку.

Если этот путь не работает целиком — MVP не готов.
